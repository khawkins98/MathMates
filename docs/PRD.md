**Version:** 2.0  
**Status:** Draft  
**Author:** Ken

## 1. Vision

MathMates is a browser-based maths game for UK Key Stage 1–2 children (ages 5–9). It lifts the grid-eating loop from *Number Munchers* and wraps it in a space-crew theme. Players navigate a grid of arithmetic expressions, eating answers that satisfy a rule to complete missions.

The goal is repeated, low-pressure arithmetic practice that feels like play, not homework — friendly enough for a five-year-old to pick up alone, varied enough to hold a nine-year-old's attention.

### 1.1 What changed from v1

v1 accumulated complexity: a pixel-art renderer tuned for near-perfect 8-bit reproduction, PNG sprite sheets, a custom GLSL post-process filter, WebGL-first PixiJS, and rendering-pipeline decisions that had to be documented at length in the README. None of that serves the educational goal.

v2 makes two course corrections:

1. **Drop the pixel-art aesthetic.** Replace it with a **sketchy, hand-drawn look** — wobbly outlines, rough fills, like a child's classroom doodle made interactive. This is warmer, more age-appropriate, and technically simpler: it is rendered entirely on a `<canvas>` with [Rough.js](https://roughjs.com), no sprite sheets, no WebGL required.

2. **Make the scenario system a first-class citizen.** v1 scenarios were functional but awkward to extend. v2 defines a clean `ScenarioDefinition` interface so any developer (or, eventually, a teacher) can add a new maths type in one small file with no changes to game code.

### 1.2 Goals

- Give children repeated, spaced arithmetic practice across a meaningful range of KS1–2 topics.
- Make adding a new scenario (e.g. "doubles", "fractions of a shape") a 15-minute task.
- Keep the technical footprint small: one static bundle, zero server, works offline.
- Deliver a visual aesthetic that feels warm and crafted, not retro-tech.

### 1.3 Non-goals (v2)

- Multiplayer or online features.
- Persistent accounts or cloud saves (session-only score tracking is fine).
- Mobile-first layout (desktop / laptop target; tablet landscape is a nice-to-have).
- Accessibility beyond keyboard navigation.

## 2. Visual Design Language

### 2.1 Sketchy / hand-drawn aesthetic

All game graphics are drawn programmatically using **Rough.js** on a Canvas 2D context. The library generates the characteristic wobbly outlines and hatch-fills that make shapes look hand-sketched. No bitmap sprites, no spritesheets, no pixel-art filters.

Design principles:
- **Outlines are alive.** Grid cells, buttons, character bodies — everything has a slightly uneven stroke, as if traced freehand.
- **Fills are textured.** Flat colour is replaced by cross-hatch or zigzag fills, lending a crayon-like warmth.
- **Characters are simple.** Crewmates and the impostor are geometric shapes (rounded rectangle body, visor ellipse, stubby legs) drawn with Rough.js. Animate them by re-drawing each frame with a small random perturbation seed — the lines "wiggle" naturally.
- **Palette is soft.** Avoid pure primaries. Use muted, slightly desaturated tones that read as "drawn with pencils" — dusty greens, warm reds, soft blues.
- **Type is legible.** Use a friendly, rounded Google Font (e.g. [Caveat](https://fonts.google.com/specimen/Caveat) for display headings, a clean sans-serif for cell values where readability under time pressure matters). Never use a font so decorative that a 5-year-old struggles with a digit.

### 2.2 Screen layout

Fixed logical canvas: **600 × 440 px**. Scaled up to fill the viewport with CSS `object-fit: contain`. Integer scaling is not required since we're not pixel art — any smooth scaling is fine.

## 3. Core Gameplay

The grid-eating loop from v1 is correct and should be kept intact. This section clarifies intent and resolves ambiguities.

### 3.1 The Grid

A **5 × 4 grid** (20 cells). Each cell holds a value generated by the active scenario (e.g. `"3 + 4"`, `"12"`, `"7 × 3"`). Cells are drawn with Rough.js rounded rectangles.

### 3.2 Player movement

Arrow keys. One cell per keypress. Wrapping at edges. The player character (crewmate or impostor depending on mode) re-draws with a small animation each move.

### 3.3 Eating a cell

Space or Enter. The cell is validated against the scenario rule:
- **Correct** → success flash, cell consumed, score updated, win check.
- **Wrong** → error flash, cell consumed, life lost, elimination sequence.

Consumed cells are visually greyed out and cannot be re-selected (except broken cells in impostor mode — see §5).

### 3.4 Marking cells as "sus"

`S` key toggles a small sketchy marker on the current cell. No gameplay effect. Clears when the cell is consumed.

### 3.5 Scoring

| Event | Points |
|---|---|
| Correct answer | +10 × streak multiplier |
| Wrong answer | multiplier resets to 1× |
| Streak multiplier | 1× → 1.5× → 2× → 3× (every 3 consecutive correct) |
| Time bonus | +50 if grid cleared within par time |
| Crewmate eliminated (impostor mode) | +25 |

## 4. Game Modes

Both modes share the same grid, scenario system, HUD layout, and briefing screen. Only the player goal and AI behaviour differ. The implementation should reflect this: a single `GameScene` parameterised by `mode: 'crew' | 'impostor'`, with no duplicated game-loop logic.

### 4.1 Crew Mode

- Player is a **crewmate**. Goal: eat every cell where `scenario.isCorrect(value) === true`.
- Hazard: an **impostor** AI wanders the grid on a random walk. If it shares a cell with the player, one life is lost and it respawns at an edge. The impostor is toggled per-stage (disabled on the easiest missions).
- Win: all correct cells consumed.
- Lose: all lives lost.

### 4.2 Impostor Mode

- Player is the **impostor**. Goal: eat every cell where `scenario.isCorrect(value) === false` (the wrong answers).
- Hazard: **AI crewmates** roam the grid looking for cells the player has broken. They repair them (restoring the cell to normal), costing the player progress but no lives. The player can eliminate a crewmate by eating a wrong cell while the crewmate stands on it.
- Win: all wrong cells consumed (and not repaired back before win-check).
- Lose: all lives lost (from accidentally eating correct cells).
- Unlocks after the player completes the same stage in crew mode.

### 4.3 Seamless mode pairing

On the stage-select screen each stage shows both a crew-mode and impostor-mode tile. The transition into impostor mode should feel like "flipping the scenario" — same briefing format, same HUD structure, different colour scheme (crew: green accent; impostor: red accent). There should be no sense that impostor mode is a bolt-on extra.

## 5. Scenario System

This is the most important architectural decision in v2. The scenario system must make it trivial to add new maths types.

### 5.1 `ScenarioDefinition` interface

```typescript
interface ScenarioDefinition {
  // Unique identifier, used in save/log keys
  id: string;

  // Human-readable strings
  title: string;        // "Multiples of 3"
  topic: string;        // "multiplication" | "addition" | "subtraction" | etc.
  ruleText: string;     // Shown on HUD: "Find multiples of 3"
  briefingText: string; // Kid-friendly: "Find every number the captain can share equally into 3 groups!"

  // Curriculum metadata
  ksYears: number[];    // e.g. [1, 2]
  difficulty: 1 | 2 | 3; // Within topic: 1 = easiest, 3 = hardest

  // Grid generation — returns exactly 20 CellValue objects
  // `seed` allows deterministic grids (useful for testing / reproducibility)
  generateGrid(seed?: number): CellValue[];

  // Rule evaluation — pure function, no side effects
  isCorrect(value: CellValue): boolean;

  // Optional: how long the player should aim to clear the grid (seconds)
  // Used for the time-bonus calculation
  parTime?: number;
}
```

`CellValue` is a small struct:

```typescript
interface CellValue {
  display: string;   // What the cell shows: "3 + 4", "12", "half of 8"
  numeric: number;   // The resolved number (for isCorrect logic)
}
```

### 5.2 Adding a new scenario

1. Create `src/scenarios/my-topic.ts`.
2. Implement `ScenarioDefinition`.
3. Register it in `src/scenarios/index.ts`.
4. Done — no other files need touching.

Example — a minimal scenario:

```typescript
// src/scenarios/doubles.ts
import type { ScenarioDefinition } from '@/types';

const doubles: ScenarioDefinition = {
  id: 'doubles-1',
  title: 'Doubles',
  topic: 'addition',
  ruleText: 'Find the doubles',
  briefingText: 'Find every number that is a double — the same number added to itself!',
  ksYears: [1],
  difficulty: 1,
  parTime: 60,
  generateGrid(seed) {
    // Generate 20 cells: ~8 doubles, ~12 non-doubles
    // Use seed for reproducibility if provided
    ...
  },
  isCorrect(value) {
    // A double is any even number between 2 and 20 for KS1
    return value.numeric % 2 === 0 && value.numeric >= 2 && value.numeric <= 20;
  },
};

export default doubles;
```

### 5.3 Stage grouping

A **Stage** is a named group of scenarios presented together on the select screen. Example: a "Times Tables" stage might contain `multiples-2`, `multiples-3`, `multiples-5`, `multiples-10` scenarios unlocked sequentially.

```typescript
interface StageDefinition {
  id: string;
  title: string;         // "Times Tables"
  description: string;   // Short blurb for the select screen
  scenarios: string[];   // Ordered scenario IDs
}
```

## 6. Technical Recommendations

### 6.1 Rendering stack

| Concern | Recommendation | Rationale |
|---|---|---|
| Canvas | Browser Canvas 2D (`CanvasRenderingContext2D`) | Rough.js targets Canvas 2D natively. No WebGL setup, no renderer abstraction. |
| Sketchy graphics | [Rough.js](https://roughjs.com) | Purpose-built for programmatic hand-drawn shapes. Zero sprites needed. |
| Build | Vite + TypeScript | Same as v1 — works well, fast HMR. |
| Font loading | Google Fonts via CSS `@import` | Caveat (headings) + a clean sans-serif (cell values). |
| Audio | Web Audio API, small synthesised tones | No external audio files needed. A thin `AudioManager` wrapper. |

Rough.js renders into a Canvas 2D context on each frame (or into an `OffscreenCanvas` for static elements). For character animation the rough seed parameter changes slightly each frame, giving the "wiggle" effect.

### 6.2 Architecture

```
src/
  core/
    GameLoop.ts        # requestAnimationFrame loop, dt clamping
    SceneManager.ts    # Scene stack, push/pop/goto
    InputManager.ts    # Keyboard queue (same pattern as v1)
  scenes/
    TitleScene.ts
    SelectScene.ts
    BriefingScene.ts
    GameScene.ts       # mode: 'crew' | 'impostor' — single unified class
    CompleteScene.ts
    GameOverScene.ts
  entities/
    Grid.ts            # Cell state machine, eat/repair/break logic
    Cell.ts            # Per-cell state + drawing
    Player.ts          # Player position + movement
    AIWanderer.ts      # Crew mode impostor AI (random walk)
    AICrewmate.ts      # Impostor mode crewmate AI (seek & repair broken cells)
  scenarios/
    index.ts           # Registry — import and export all scenarios
    addition.ts
    subtraction.ts
    multiplication.ts
    division.ts
    # ... one file per topic group
  rendering/
    RoughRenderer.ts   # Thin wrapper: draw cell, draw character, draw button
    colours.ts         # Palette constants
  audio/
    AudioManager.ts
  ui/
    HUD.ts
    Button.ts
    BriefingOverlay.ts
    EliminationOverlay.ts
  types.ts
  constants.ts
```

### 6.3 Game loop notes

- Target 60 fps via `requestAnimationFrame`.
- Clamp `dt` to 100 ms to prevent spiral-of-death on tab switch / slow devices.
- Re-draw the entire canvas each frame (Canvas 2D `clearRect`). With 20 cells and a few characters this is trivially fast.
- Static elements (background, grid outlines) can be cached into an `OffscreenCanvas` and composited as a single `drawImage` call to reduce Rough.js recalculation.

## 7. Stage Roadmap (v2 launch)

| Stage | Topic | Difficulty | KS Year |
|---|---|---|---|
| Counting On | Addition (single digit) | 1 | KS1 |
| Taking Away | Subtraction (single digit) | 1 | KS1 |
| Two Times | Multiples of 2 | 1 | KS1 |
| Five Times | Multiples of 5 | 1 | KS1 |
| Ten Times | Multiples of 10 | 1 | KS1 |
| Adding to 20 | Addition (to 20) | 2 | KS1–2 |
| Taking from 20 | Subtraction (from 20) | 2 | KS1–2 |
| Three Times | Multiples of 3 | 2 | KS2 |
| Four Times | Multiples of 4 | 2 | KS2 |
| Mixed Tables | Multiples of 2, 5, 10 mixed | 3 | KS2 |

Each stage has 3 missions of increasing grid density / number range.

## 8. Open Questions

- **Accessibility**: Should the game support a pointer/touch control scheme, or remain keyboard-only for v2?
- **Localisation**: English-only for v2, but does the scenario `briefingText` need to be internationalisation-ready?
- **Persistence**: Session-only scoring is fine for v2, but should the stage-unlock state survive a page reload (localStorage)?
- **Rough.js animation strategy**: Per-frame re-seed gives organic wiggle but costs CPU. Benchmark on target hardware (Chromebook) early.
